11:T273b,<h1>API Schema Contracts in a Full-Stack Monorepo</h1>
<p>A PR I recently read through included instructions for a Node.js BFF (Backend for Frontend) layer that stood out for how explicitly it defined the contract between frontend and backend. Three principles in particular are worth breaking down.</p>
<hr>
<h2>What Is a BFF?</h2>
<p>A BFF (Backend for Frontend) is a thin server layer that sits between the frontend app and the real backend APIs. Its job is to:</p>
<ul>
<li>Aggregate and reshape data from multiple backend services into what the UI actually needs</li>
<li>Validate and sanitize frontend inputs before they reach internal services</li>
<li>Own the frontend's API surface so the UI doesn't couple directly to backend internals</li>
</ul>
<p>The key word in all three points is <em>thin</em>. A BFF is not a place to put business logic.</p>
<hr>
<h2>Principle 1: Keep Routes Thin</h2>
<p>The guideline from the PR was direct:</p>
<blockquote>
<p>Keep BFF routes thin: validate/shape inputs at the edge and delegate behavior to services.</p>
</blockquote>
<p>In practice, this means a route handler should do two things and nothing else: validate the incoming request shape, then call a service. All logic lives in the service layer.</p>
<pre><code class="hljs language-ts"><span class="hljs-comment">// Thin route — validates at the edge, delegates to service</span>
router.get(<span class="hljs-string">&quot;/users/:id/profile&quot;</span>, <span class="hljs-keyword">async</span> (req, res) =&gt; {
  <span class="hljs-keyword">const</span> { id } = validateParams(req.params, userIdSchema);
  <span class="hljs-keyword">const</span> profile = <span class="hljs-keyword">await</span> userProfileService.getProfile(id);
  res.json(profile);
});

<span class="hljs-comment">// Fat route — logic mixed into the handler (avoid this)</span>
router.get(<span class="hljs-string">&quot;/users/:id/profile&quot;</span>, <span class="hljs-keyword">async</span> (req, res) =&gt; {
  <span class="hljs-keyword">const</span> id = req.params.id;
  <span class="hljs-keyword">if</span> (!id || <span class="hljs-built_in">isNaN</span>(<span class="hljs-built_in">Number</span>(id))) {
    <span class="hljs-keyword">return</span> res.status(<span class="hljs-number">400</span>).json({ <span class="hljs-attr">error</span>: <span class="hljs-string">&quot;Invalid ID&quot;</span> });
  }
  <span class="hljs-keyword">const</span> raw = <span class="hljs-keyword">await</span> db.query(<span class="hljs-string">&quot;SELECT * FROM users WHERE id = ?&quot;</span>, [id]);
  <span class="hljs-keyword">const</span> profile = {
    <span class="hljs-attr">name</span>: <span class="hljs-string">`<span class="hljs-subst">${raw.firstName}</span> <span class="hljs-subst">${raw.lastName}</span>`</span>,
    <span class="hljs-attr">email</span>: raw.email,
  };
  res.json(profile);
});
</code></pre>
<p>The thin version is easier to test (the service is independently testable), easier to read, and makes the route's responsibility obvious at a glance.</p>
<hr>
<h2>Principle 2: Explicit Typed Contracts</h2>
<blockquote>
<p>Prefer explicit typed contracts for request/response shapes over ad hoc object handling.</p>
</blockquote>
<p>This is about treating the API boundary like a typed interface, not a loosely-shaped JSON blob. Every route should have an explicit type for what it accepts and what it returns.</p>
<pre><code class="hljs language-ts"><span class="hljs-comment">// Explicit contracts</span>
<span class="hljs-keyword">interface</span> GetUserProfileParams {
  <span class="hljs-attr">id</span>: <span class="hljs-built_in">string</span>;
}

<span class="hljs-keyword">interface</span> UserProfileResponse {
  <span class="hljs-attr">firstName</span>: <span class="hljs-built_in">string</span>;
  lastName: <span class="hljs-built_in">string</span>;
  email: <span class="hljs-built_in">string</span>;
  role: <span class="hljs-string">&quot;admin&quot;</span> | <span class="hljs-string">&quot;member&quot;</span> | <span class="hljs-string">&quot;guest&quot;</span>;
}

<span class="hljs-keyword">async</span> <span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">getProfile</span>(<span class="hljs-params">params: GetUserProfileParams</span>): <span class="hljs-title">Promise</span>&amp;#<span class="hljs-title">x3C</span></span>;UserProfileResponse&gt; {
  <span class="hljs-comment">// ...</span>
}
</code></pre>
<p>Without this, the frontend team is left guessing at response shapes from the network tab. With explicit types shared between the BFF and the React app (via a shared types package or generated schema), both sides have a contract to code against.</p>
<p>The benefit compounds in a monorepo: when the BFF type changes, TypeScript surfaces every place the frontend needs to update.</p>
<hr>
<h2>Principle 3: Feature Flags as an API Contract</h2>
<p>The PR also described how feature flags are exposed:</p>
<blockquote>
<p>Flags are BFF-driven and exposed at <code>GET /api/features</code> using OpenFeature-style schema.
Env key convention: <code>APP_FEATURE_&#x3C;FLAG_NAME></code> → UI key <code>flag_name</code>.</p>
</blockquote>
<p>This is a clean pattern. Rather than each frontend component reading environment variables directly (which doesn't work in a browser anyway), the BFF owns the feature flag state and exposes it through a single typed endpoint.</p>
<pre><code class="hljs language-ts"><span class="hljs-comment">// BFF exposes flags via a typed response</span>
<span class="hljs-keyword">interface</span> FeatureFlagsResponse {
  <span class="hljs-attr">dark_mode</span>: <span class="hljs-built_in">boolean</span>;
  new_dashboard: <span class="hljs-built_in">boolean</span>;
  beta_export: <span class="hljs-built_in">boolean</span>;
}

router.get(<span class="hljs-string">&quot;/api/features&quot;</span>, <span class="hljs-function">(<span class="hljs-params">_req, res</span>) =&gt;</span> {
  <span class="hljs-keyword">const</span> flags: FeatureFlagsResponse = {
    <span class="hljs-attr">dark_mode</span>: process.env.APP_FEATURE_DARK_MODE === <span class="hljs-string">&quot;true&quot;</span>,
    <span class="hljs-attr">new_dashboard</span>: process.env.APP_FEATURE_NEW_DASHBOARD === <span class="hljs-string">&quot;true&quot;</span>,
    <span class="hljs-attr">beta_export</span>: process.env.APP_FEATURE_BETA_EXPORT === <span class="hljs-string">&quot;true&quot;</span>,
  };
  res.json(flags);
});
</code></pre>
<p>The frontend fetches this once at startup and uses the typed response. The naming convention (<code>APP_FEATURE_&#x3C;FLAG_NAME></code> → <code>flag_name</code>) makes it predictable to add new flags without changing the pattern.</p>
<hr>
<h2>Principle 4: Safe Error Handling</h2>
<blockquote>
<p>Keep route and service error handling predictable and user-safe — no leaking internal stack details to the client.</p>
</blockquote>
<p>This one is security hygiene. Stack traces, database error messages, and internal service names should never reach the browser. The BFF should translate errors into safe, predictable shapes:</p>
<pre><code class="hljs language-ts"><span class="hljs-comment">// Safe error handler — maps internal errors to user-safe responses</span>
app.use(<span class="hljs-function">(<span class="hljs-params">err: <span class="hljs-built_in">Error</span>, _req: Request, res: Response, _next: NextFunction</span>) =&gt;</span> {
  <span class="hljs-built_in">console</span>.error(err); <span class="hljs-comment">// log internally</span>

  <span class="hljs-keyword">if</span> (err <span class="hljs-keyword">instanceof</span> ValidationError) {
    <span class="hljs-keyword">return</span> res.status(<span class="hljs-number">400</span>).json({ <span class="hljs-attr">error</span>: err.message });
  }

  <span class="hljs-keyword">if</span> (err <span class="hljs-keyword">instanceof</span> NotFoundError) {
    <span class="hljs-keyword">return</span> res.status(<span class="hljs-number">404</span>).json({ <span class="hljs-attr">error</span>: <span class="hljs-string">&quot;Not found&quot;</span> });
  }

  <span class="hljs-comment">// Default: don&#x27;t expose internal details</span>
  res.status(<span class="hljs-number">500</span>).json({ <span class="hljs-attr">error</span>: <span class="hljs-string">&quot;Something went wrong&quot;</span> });
});
</code></pre>
<p>The internal error is logged for debugging. The client gets a safe, predictable shape.</p>
<hr>
<h2>Putting It Together</h2>
<p>These four principles form a coherent architecture for the BFF layer:</p>
<table><thead><tr><th>Principle</th><th>What it prevents</th></tr></thead><tbody><tr><td>Thin routes</td><td>Business logic scattered across handlers</td></tr><tr><td>Typed contracts</td><td>Implicit coupling between frontend and backend</td></tr><tr><td>Feature flags as API</td><td>Environment variables leaking into frontend bundles</td></tr><tr><td>Safe error handling</td><td>Internal stack details reaching the client</td></tr></tbody></table>
<p>The common thread: the BFF is a boundary. Everything that crosses that boundary — requests in, responses out, errors, feature state — should be explicitly shaped and typed. Ad hoc handling at the boundary is where full-stack systems quietly accumulate technical debt.</p>
<hr>
<h2>Takeaways</h2>
<ul>
<li>A BFF should validate at the edge and delegate everything else to services. If your route handler has business logic in it, that's a smell.</li>
<li>Typed request/response contracts pay off most in a monorepo where frontend and backend share types — TypeScript surfaces breakage immediately.</li>
<li>Exposing feature flags through a single BFF endpoint is cleaner than reading env vars on the frontend, and the naming convention (<code>APP_FEATURE_X</code> → <code>x</code>) keeps it mechanical to extend.</li>
<li>Error handling at the BFF boundary should be a deliberate design decision, not an afterthought.</li>
</ul>
