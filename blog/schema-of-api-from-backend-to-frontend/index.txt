11:T3aea,<p>If you've worked across backend and frontend, you've probably run into this:</p>
<blockquote>
<p>Backend renames a field. Frontend doesn't update. Everything compiles... until it silently breaks at runtime.</p>
</blockquote>
<p>This post walks through how to fix that <strong>systematically</strong> — not just patch it case by case.</p>
<hr>
<h2>The Core Problem: Contract Drift</h2>
<p>When backend and frontend are developed separately, it's easy for the data contract to drift:</p>
<ul>
<li>Backend renames <code>personalNumber</code> → <code>personalPhone</code></li>
<li>Frontend still uses <code>personalNumber</code></li>
<li>No compile error, no immediate warning</li>
<li>Users see broken UI or empty fields</li>
</ul>
<p>The deeper issue: <strong>frontend was manually maintaining its own type definitions</strong> instead of deriving them from the actual backend contract.</p>
<hr>
<h2>The Solution: Schema-Driven Workflow</h2>
<p>The idea is simple:</p>
<p><strong>Backend → OpenAPI Schema → Generated Types → Frontend</strong></p>
<p>Instead of writing TypeScript interfaces by hand and hoping they match, you:</p>
<ol>
<li>Let the backend <strong>generate</strong> an OpenAPI schema</li>
<li>Use tooling to <strong>generate TypeScript types</strong> from that schema</li>
<li>Frontend <strong>imports</strong> those generated types</li>
</ol>
<p>Now when backend changes, the schema changes, the types change, and TypeScript <strong>tells you</strong> what broke — at compile time, not at runtime.</p>
<hr>
<h2>Why OpenAPI Schema Is More Than Just Documentation</h2>
<p>A lot of developers treat Swagger/OpenAPI as just a documentation page at <code>/swagger</code>. That mental model undersells it.</p>
<p>Think of the OpenAPI schema as a <strong>formal contract</strong> between backend and frontend:</p>
<ul>
<li>It defines every endpoint, request shape, and response shape</li>
<li>It's machine-readable</li>
<li>It can be used to generate code</li>
<li>It can be validated automatically in CI</li>
</ul>
<p>Once you treat it as a contract, you can build tooling around it.</p>
<hr>
<h2>The Workflow, Step by Step</h2>
<h3>1. Backend publishes an OpenAPI schema</h3>
<p>With ASP.NET Core (C#), enabling Swagger is straightforward. The same API that serves your business data also exposes:</p>
<ul>
<li>Swagger UI: <code>/swagger</code></li>
<li>Schema JSON: <code>/swagger/v1/swagger.json</code></li>
</ul>
<p>No separate service needed. Your existing API container can serve both.</p>
<h3>2. Commit the schema snapshot</h3>
<p>Check the generated <code>openapi.json</code> into your repository. This snapshot serves as:</p>
<ul>
<li>The input for frontend type generation</li>
<li>A reviewable diff when the contract changes</li>
<li>The baseline for automated schema sync checks</li>
</ul>
<h3>3. Generate TypeScript types from the schema</h3>
<p>A common tool for this is <a href="https://github.com/openapi-ts/openapi-typescript"><code>openapi-typescript</code></a>:</p>
<pre><code class="hljs language-bash">npm install --save-dev openapi-typescript

<span class="hljs-comment"># From a running backend</span>
npx openapi-typescript http://localhost:5000/swagger/v1/swagger.json -o src/api/generated/schema.ts

<span class="hljs-comment"># Or from a committed schema file</span>
npx openapi-typescript ./openapi.json -o src/api/generated/schema.ts
</code></pre>
<h3>4. Use generated types in frontend code</h3>
<pre><code class="hljs language-ts"><span class="hljs-keyword">import</span> <span class="hljs-keyword">type</span> { paths } <span class="hljs-keyword">from</span> <span class="hljs-string">&quot;./api/generated/schema&quot;</span>;

<span class="hljs-keyword">type</span> ContactInfo =
  paths[<span class="hljs-string">&quot;/employees/{employeeId}/contact&quot;</span>][<span class="hljs-string">&quot;get&quot;</span>][<span class="hljs-string">&quot;responses&quot;</span>][<span class="hljs-number">200</span>][<span class="hljs-string">&quot;content&quot;</span>][<span class="hljs-string">&quot;application/json&quot;</span>];

<span class="hljs-keyword">export</span> <span class="hljs-keyword">async</span> <span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">fetchContactInfo</span>(<span class="hljs-params">employeeId: <span class="hljs-built_in">string</span></span>): <span class="hljs-title">Promise</span>&amp;#<span class="hljs-title">x3C</span></span>;ContactInfo&gt; {
  <span class="hljs-keyword">const</span> response = <span class="hljs-keyword">await</span> axios.get&amp;#x3C;ContactInfo&gt;(<span class="hljs-string">`/employees/<span class="hljs-subst">${employeeId}</span>/contact`</span>);
  <span class="hljs-keyword">return</span> response.data;
}
</code></pre>
<p>No more guessing the shape. No more hand-written interfaces. The types come directly from the backend definition.</p>
<h3>5. Enforce sync in CI</h3>
<p>Add an integration test that:</p>
<ul>
<li>Reads the committed <code>openapi.json</code></li>
<li>Requests the live <code>/swagger/v1/swagger.json</code> from the running API</li>
<li>Fails if they differ</li>
</ul>
<p>This makes it <strong>impossible to forget</strong> updating the schema. If someone changes a backend model and doesn't regenerate the schema, CI catches it before merge.</p>
<hr>
<h2>What About Separate Repos?</h2>
<p>You don't need frontend and backend in the same repo for this to work. The key is that:</p>
<ol>
<li><strong>Backend publishes the schema</strong> — either as a committed file, a CI artifact, or via a URL</li>
<li><strong>Frontend consumes it</strong> — by downloading and regenerating types</li>
</ol>
<p>Common publishing approaches in separate-repo setups:</p>
<table><thead><tr><th>Approach</th><th>Description</th></tr></thead><tbody><tr><td>CI artifact</td><td>Backend CI generates and uploads <code>openapi.json</code> as a build artifact</td></tr><tr><td>Object storage</td><td>Published to S3/blob storage after each backend build</td></tr><tr><td>Shared contract repo</td><td>A dedicated repo that only holds the schema file</td></tr><tr><td>Direct URL</td><td>Frontend CI downloads from the running backend's <code>/swagger/v1/swagger.json</code></td></tr></tbody></table>
<p>The workflow stays the same. Only the distribution step changes.</p>
<hr>
<h2>Naming Matters More Than You Think</h2>
<p>Part of what this workflow makes visible: <strong>field naming is a public API decision</strong>.</p>
<p>Renaming <code>personalNumber</code> to <code>personalPhone</code> isn't just a code style change — it changes the contract. With schema-driven types, that rename shows up in:</p>
<ul>
<li>The OpenAPI schema diff</li>
<li>The generated TypeScript types</li>
<li>Any frontend code using the old name (now a compile error)</li>
</ul>
<p>This is a feature. It makes contract changes explicit and reviewable.</p>
<h3>Decoupling internal names from external contract</h3>
<p>Sometimes you want to refactor internal class or field names without breaking the external API. In C# with Swashbuckle, you can use the <code>[JsonPropertyName]</code> attribute or configure Swagger to use a different name:</p>
<pre><code class="hljs language-csharp"><span class="hljs-keyword">using</span> System.Text.Json.Serialization;

<span class="hljs-keyword">public</span> <span class="hljs-keyword">class</span> <span class="hljs-title">ContactInfoView</span>
{
    [<span class="hljs-meta">JsonPropertyName(<span class="hljs-meta-string">&quot;workPhone&quot;</span>)</span>]
    <span class="hljs-keyword">public</span> <span class="hljs-built_in">string</span>? WorkPhoneInternal { <span class="hljs-keyword">get</span>; <span class="hljs-keyword">set</span>; }
}
</code></pre>
<p>This way the internal property name can change, but the API contract stays stable.</p>

<hr>
<h2>Key Takeaways</h2>
<ul>
<li><strong>Backend is the source of truth.</strong> Frontend should never guess data shapes.</li>
<li><strong>Generate types, don't write them.</strong> If a schema exists, use it.</li>
<li><strong>CI can enforce architecture, not just business logic.</strong> Schema sync tests prevent drift automatically.</li>
<li><strong>Naming is a contract decision.</strong> Explicit names reduce bugs and improve readability.</li>
<li><strong>You don't need a separate schema service.</strong> Your existing backend API can expose the schema via Swagger.</li>
</ul>
<hr>
<h2>Before vs. After</h2>
<table><thead><tr><th>Before</th><th>After</th></tr></thead><tbody><tr><td>Frontend hand-writes type interfaces</td><td>Types generated from schema</td></tr><tr><td>Backend and frontend drift silently</td><td>Schema sync test fails in CI</td></tr><tr><td>Rename = hope frontend stays in sync</td><td>Rename = compile error guides you</td></tr><tr><td>Duplicated type definitions</td><td>Single source of truth</td></tr><tr><td>Runtime surprises</td><td>Compile-time errors</td></tr></tbody></table>
<p>The mental shift is:</p>
<blockquote>
<p>From "keeping frontend and backend in sync manually" to "letting the contract enforce consistency automatically."</p>
